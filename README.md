**二、RPA / 机器人任务调度**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 7 | 讲一下你在金蝶 RPA 平台从 0 到 1 做了什么。 | 负责云端控制台，包括流程管理、机器人管理、任务调度、执行监控、Open API、许可控制。核心是把流程、任务、机器人、状态、日志抽象成平台能力。 |
| 8 | RPA 任务调度和机器人任务调度有什么相似和不同？ | 相似：都有任务、执行资源、状态、日志、重试。不同：实体机器人多了物理状态、安全、实时性、离线恢复、指令幂等和云边通信。 |
| 9 | 机器人任务调度系统怎么设计状态机？ | 设计 CREATED、WAITING、DISPATCHED、RUNNING、SUCCESS、FAILED、CANCELED、TIMEOUT 等状态。状态流转必须受控，并记录变更日志。 |
| 10 | RPA 平台里机器人在线 / 离线状态怎么判断？ | 通过心跳、连接状态、最后上报时间判断。超过阈值未上报标记离线，恢复心跳后重新同步机器人状态和任务上下文。 |
| 11 | RPA 客户端执行中崩溃，云端怎么处理？ | 通过心跳超时或任务回调超时感知。任务进入异常或待确认状态，结合任务是否可重试决定自动重试、人工处理或标记失败。 |
| 12 | 如果机器人离线了，任务怎么办？ | 未下发任务可以重新调度；已下发未 ACK 标记下发超时；执行中离线要等设备恢复后同步状态，再决定继续、取消或人工介入。 |
| 13 | 多机器人协同执行一个任务，怎么设计？ | 把大任务拆成子任务，给每个机器人分配资源和角色。父任务汇总子任务状态，处理部分失败、重试、取消和整体完成判断。 |
| 14 | AI 训练任务和普通后台任务有什么区别？ | AI 训练任务通常耗时长、资源重、依赖 GPU、需要排队、可中断恢复、需要保存模型版本和指标。普通后台任务更偏业务处理和短事务。 |

**三、GPT / AI 平台**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 15 | GPT 平台里模型统一接入怎么设计？ | 做统一模型调用接口，上层屏蔽厂商差异。底层用适配器模式对接不同模型，处理鉴权、参数转换、响应解析、异常映射和流式返回。 |
| 16 | GPT 平台 Token 计费怎么设计？ | 核心对象有租户、模型实例、调用任务、用量记录、扣减流水。通过 taskId 保证幂等，调用记录和计费流水分开，方便审计和追踪。 |
| 17 | 不同模型厂商响应慢、限流、失败，怎么处理？ | 设置超时、重试、熔断、降级和限流策略。错误分可重试和不可重试，模型调用记录要完整保存，便于监控和排查。 |
| 18 | GPT 流式输出后端怎么设计？ | 后端用 SSE 或 WebSocket 返回流式内容。服务端边接收模型流边转发给前端，同时记录完整任务状态、异常和最终结果。 |
| 19 | 你怎么理解 MLOps？ | MLOps 是模型从数据、训练、评估、发布、监控到回滚的工程化闭环。机器人场景重点是数据回流、模型版本、灰度发布和效果监控。 |
| 20 | 如果模型服务调用失败，业务侧怎么感知？ | 平台统一返回标准错误码和任务状态，业务侧不直接感知厂商细节。调用链路记录请求、响应、耗时、错误原因，便于追踪。 |

**四、IOT / 云边通信 / 设备接入**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 21 | 你在云鲸 IOT 平台具体做过什么？ | 参与设备管理、设备接入、MQTT 消息处理、数据采集、指令下发、告警处理和问题排查。重点讲设备上报、解析、落库、远程控制闭环。 |
| 22 | 云端给机器人下发指令，怎么保证可靠和低延迟？ | 使用 commandId 做幂等，指令状态包含已创建、已下发、已 ACK、执行中、成功、失败、超时。低延迟靠长连接、MQTT/gRPC/WebSocket 和减少同步依赖。 |
| 23 | 指令已下发，但机器人迟迟不回结果，怎么办？ | 区分未 ACK、已 ACK 未完成、执行超时。通过超时时间、心跳、设备状态和任务上下文判断，必要时重试或人工介入。 |
| 24 | 什么是设备影子 Device Shadow？ | 设备影子是云端保存的一份设备状态副本，包括期望状态和上报状态。设备离线时，云端可以保存期望状态，设备上线后再同步。 |
| 25 | 设备高频上报重复数据或脏数据，怎么处理？ | 入口做校验、去重、限流和异常过滤。重复数据用消息 ID、时间戳、版本号判断，脏数据进入异常日志或死信队列。 |
| 26 | 如果要支持 10 万台机器人同时在线，关注哪些瓶颈？ | 重点关注连接数、消息吞吐、Broker 扩展、分区策略、状态存储、数据库写入、热点设备、监控告警和水平扩容能力。 |
| 27 | 机器人远程升级固件或模型，怎么设计？ | 需要版本管理、灰度发布、分批升级、完整性校验、失败重试、回滚机制和升级状态追踪。危险版本不能全量直接推。 |
| 28 | 机器人平台如何做安全控制？ | 设备认证、通信加密、权限校验、指令白名单、危险操作拦截、审计日志、固件签名和异常行为告警。 |

**五、机器人 / AI 云平台架构设计**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 29 | 如果让你设计机器人 / AI 云平台，你会怎么做？ | 分设备接入层、任务平台层、数据链路层、AI/MLOps 层、管理后台层。核心是可靠接入、任务可追踪、数据可扩展、模型可发布回滚。 |
| 30 | 大量机器人状态数据上报，怎么保证系统扛得住？ | 入口用 MQTT/MQ 削峰，实时状态放 Redis 或状态表，历史数据进时序库/OLAP/对象存储。消费者异步处理、批量写入、失败重试。 |
| 31 | 机器人日志、传感器数据、图片视频分别怎么存？ | 普通日志进日志系统，传感器时序数据进时序库或 OLAP，图片视频进对象存储，数据库只保存索引、元数据和关联关系。 |
| 32 | 机器人平台可观测性怎么做？ | 设备侧看在线率、心跳、指令延迟；任务侧看排队、执行、失败率；系统侧看 QPS、P99、MQ 堆积、慢 SQL。用 taskId、deviceId、commandId 串链路。 |
| 33 | 数据链路怎么设计？ | 实时数据走消息队列或流处理，历史数据进入数仓/数据湖，关键状态进业务库或缓存。不同数据按实时监控、离线分析、训练数据分层。 |
| 34 | 云平台如何支持仿真、训练、部署任务？ | 抽象统一任务模型，不同任务有不同执行器。仿真/训练适合 K8s Job，部署任务要支持版本、灰度、回滚和执行结果追踪。 |

**六、低代码 / 流程引擎 / 业务建模**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 35 | 低代码表单与流程引擎核心难点是什么？ | 难点是动态表单存储、流程状态机、版本兼容、权限控制和业务扩展。不能只存 JSON，要考虑查询、统计和审计。 |
| 36 | 流程发布后字段或节点变更，历史实例怎么兼容？ | 流程定义和表单定义要有版本。历史实例绑定旧版本，新实例使用新版本，避免新配置影响已运行流程。 |
| 37 | EAM 资产生命周期怎么建模？ | 围绕申请、审批、采购、入库、出库、验收、建账、转交、回收等状态建模。核心是状态机、操作记录和资产台账可追溯。 |
| 38 | SCM 中台规则计算怎么设计？ | 规则配置、规则执行、执行结果和审计记录分开。规则执行要可追踪、可回放，失败要有日志和补偿机制。 |

**七、后端基础与分布式**

| # | 面试题 | 简易答案 / 核心思路 |
|---|---|---|
| 39 | MQ 如何处理重复消费和消息顺序？ | 重复消费靠业务幂等，消息带 eventId/taskId。顺序只保证同一设备或同一任务内有序，用 key 分区或同队列消费。 |
| 40 | 接口幂等怎么设计？ | 用 requestId、taskId、commandId 做唯一键。服务端保存处理结果，重复请求直接返回已有结果，避免重复创建、重复扣费、重复下发。 |
| 41 | 高可靠任务重试机制怎么设计？ | 区分可重试和不可重试错误，设置最大重试次数、退避策略、死信队列和告警。重试必须配合幂等。 |
| 42 | 分布式锁用在哪些场景？怎么避免问题？ | 用于防止并发调度、重复执行、定时任务抢占。锁要设置过期时间，value 唯一，释放时校验 value，避免误删别人的锁。 |
| 43 | 如何做接口限流？ | 可以按用户、租户、设备、IP、接口维度限流。核心接口可用令牌桶/漏桶，超限返回明确错误，保护后端和第三方服务。 |
| 44 | 跨系统数据一致性怎么保证？ | 不做长事务，采用状态机、调用记录、重试、补偿和对账。外部调用必须记录请求、响应、状态和错误原因。 |
| 45 | MySQL 性能优化怎么做？ | 先看慢 SQL 和执行计划，再优化索引、分页、批量写、表结构和冷热数据。大数据量日志/状态数据考虑归档或 OLAP。 |
| 46 | Redis 常用场景和坑？ | 用于缓存、状态快照、分布式锁、限流、幂等记录。注意缓存穿透、击穿、雪崩、大 key、热 key 和一致性问题。 |
| 47 | 线上接口 P99 延迟升高，你怎么排查？ | 先看监控定位是接口、数据库、MQ、缓存、下游服务还是机器资源。再查慢 SQL、线程池、GC、连接池、日志和最近发布。 |
| 48 | K8s / Helm / Operator 你怎么理解？ | K8s 做容器编排，Helm 做模板化部署，Operator 把复杂运维自动化。可以承认不是最深，但理解其在训练、仿真、部署任务里的价值。 |

**建议你重点背这 5 个万能答题框架**

1. **任务调度类**：任务模型、状态机、资源匹配、幂等、重试、超时、日志、监控。

2. **设备通信类**：设备认证、心跳、MQTT/gRPC、ACK、指令状态、离线恢复、安全校验。

3. **AI 平台类**：模型适配器、任务分发、同步/异步/流式、调用监控、Token 统计、限流熔断。

4. **数据链路类**：实时数据、历史数据、日志数据、对象存储、OLAP、数据湖、可追踪。

5. **短板应对类**：不夸大 Go/K8s/机器人经验，强调 Java 后端能力、RPA/GPT/IOT 经验可迁移，并说明学习路径。
