**8. GPT 平台里的 Token 计费你怎么设计？**

可以这样答：

我会把计费拆成几个核心对象：租户或客户、模型实例、调用任务、用量记录、资源包或额度、扣减流水。

调用链路上，模型调用完成后拿到输入输出 Token 或厂商返回的用量信息，生成一条不可变的用量流水。扣减时要保证幂等，通常会用任务 ID 或调用 ID 做唯一键，避免重试导致重复扣费。

计费还要考虑几类异常：调用失败是否计费、流式调用中断怎么统计、厂商未返回 Token 怎么兜底、扣减失败是否补偿。我的倾向是调用记录和扣费流水分开，调用记录反映真实调用过程，扣费流水反映财务口径，二者通过任务 ID 关联，方便审计和问题追踪。

**三、系统设计高频题**

**11. 如果让你设计一个机器人 / AI 云平台，你会怎么做整体架构？**

可以这样答：

我会分成几层设计。

第一层是设备接入层，负责机器人注册、认证、心跳、状态上报、日志上报、指令下发，可以支持 MQTT、gRPC、WebSocket、HTTP 等协议，根据实时性和场景选择。

第三层是数据链路层，处理机器人感知数据、控制数据、状态数据和日志数据。实时数据走消息队列或流式处理，历史数据进入对象存储、时序库、OLAP 或数据湖，供监控、分析和模型训练使用。

第四层是 AI / MLOps 层，管理模型、数据集、训练任务、评估指标、版本发布、灰度部署和回滚。

第五层是平台管理层，包括租户、权限、资源配额、审计、告警、可观测性和后台管理系统。

整体原则是：设备接入要可靠，任务调度要可追踪，数据链路要可扩展，模型部署要可回滚，所有关键链路都要可观测。

**12. 机器人大量状态数据上报，你怎么保证系统扛得住？**

可以这样答：

我会从入口削峰、数据分层和存储选型三方面处理。

入口层面，设备上报不要直接打数据库，先进入 MQTT Broker 或消息队列，后面用消费者异步处理。按设备 ID、租户、数据类型做分区，避免单分区热点。

数据分层方面，实时状态只保留最新值，适合放 Redis 或状态表；历史轨迹、传感器数据、日志数据进入时序库、对象存储、ClickHouse / Doris 这类 OLAP；原始大文件比如图片、点云、视频片段放对象存储，只在数据库保存索引和元数据。

处理链路上要做批量写入、异步落库、失败重试、死信队列和数据质量校验。对于高频数据，不是所有字段都要全量入库，可以根据业务价值区分实时监控、离线分析和训练数据。

**13. 云端给机器人下发指令，怎么保证可靠和低延迟？**

可以这样答：

我会把指令下发设计成“有状态、有确认、可追踪”的链路。

云端生成指令时创建唯一 commandId，记录目标设备、指令内容、版本号、超时时间和状态。下发后设备先回收到 ACK，再执行指令，执行完成后回 result。云端根据 ACK 和 result 更新状态。

可靠性上，要处理重复下发、网络抖动、设备离线、超时和乱序。可以通过 commandId 做幂等，通过 sequence 或版本号处理乱序，通过 TTL 避免旧指令被执行，通过重试和死信处理失败。

低延迟上，协议可以选 MQTT、WebSocket 或 gRPC 长连接，关键链路减少同步依赖，不要在下发路径里做重计算。对于危险指令，还要有权限校验、设备状态校验和安全拦截，比如机器人不在线、电量不足、处于异常状态时不能执行某些动作。

---

**14. 机器人任务调度系统你会怎么设计状态机？**

可以这样答：

我会把任务状态设计清楚，避免后面逻辑混乱。常见状态包括：CREATED、WAITING、DISPATCHED、RUNNING、PAUSED、SUCCESS、FAILED、CANCELED、TIMEOUT。

状态流转必须受控，不能任意更新。比如 CREATED 可以进入 WAITING，WAITING 可以 DISPATCHED 或 CANCELED，RUNNING 可以 SUCCESS、FAILED、TIMEOUT 或 PAUSED。每次状态变化都记录状态变更日志，包括操作人、触发来源、原因、时间和上下文。

调度层面还要考虑优先级、资源占用、机器人能力匹配、任务互斥、重试次数、超时策略和人工干预。对于分布式部署，任务领取要有锁或 CAS 机制，避免多个 worker 同时调度同一个任务。

**15. 如果机器人离线了，任务怎么办？**

可以这样答：

要看任务类型和执行阶段。

如果任务还没下发，机器人离线后应该重新进入待调度状态，或者调度给其他满足条件的机器人。如果任务已下发但还没 ACK，可以标记为下发超时，允许重试或人工处理。如果任务执行中机器人离线，要根据业务判断是否可恢复。

我会设计任务检查点和设备心跳机制。设备恢复在线后，云端先同步设备当前状态和任务上下文，再决定是继续任务、取消任务还是重新下发。这里最重要的是不能盲目重复执行，特别是涉及物理动作的任务，必须通过 commandId、任务版本和设备当前状态做幂等和安全校验。

**16. 怎么做机器人平台的可观测性？**

可以这样答：

我会分三类指标。

第一是设备侧指标：在线数、离线数、心跳延迟、消息上报频率、指令 ACK 延迟、执行成功率、异常告警数量。

第二是任务侧指标：任务创建量、排队时长、调度耗时、执行时长、成功率、失败原因分布、重试次数、超时任务数。

第三是系统侧指标：接口 QPS、P95/P99 延迟、MQ 堆积、消费者延迟、数据库慢 SQL、错误率、服务资源使用率。

日志方面要有 traceId、deviceId、taskId、commandId，把一次任务从创建、调度、下发、ACK、执行、回调、完成串起来。否则出了问题，只看单个服务日志很难定位。

**四、后端技术与工程能力**

**19. RabbitMQ / MQ 使用中如何处理重复消费和消息顺序？**

可以这样答：

重复消费不能只依赖 MQ 本身完全避免，业务侧必须做幂等。常见方式是消息里带唯一业务 ID，比如 taskId、commandId、eventId，消费端用唯一索引、状态判断或幂等表保证同一事件只生效一次。

顺序问题要看业务是否真的要求全局有序。多数场景只需要同一个设备、同一个任务内有序，可以按 deviceId 或 taskId 分区，保证同一 key 进入同一队列或同一消费者。对于乱序到达的状态回调，可以用状态机和版本号控制，低版本状态不能覆盖高版本状态。

失败处理上，我会设置重试次数、延迟重试、死信队列和告警，避免坏消息一直阻塞正常消费。

**20. 你怎么做接口幂等？**

可以这样答：

要先看接口类型。查询接口天然幂等，创建、支付、扣费、指令下发、任务启动这类接口必须重点设计。

常见方案是客户端或服务端生成幂等键，比如 requestId、taskId、commandId。服务端用唯一索引或幂等记录表保存请求处理结果。重复请求进来时，如果前一次成功，就直接返回之前的结果；如果处理中，就返回处理中或等待；如果失败，要根据失败类型决定是否允许重试。

在机器人指令下发场景，幂等尤其重要，因为重复执行可能造成物理风险。所以除了接口层幂等，还要在设备侧识别 commandId，已经执行过的指令不能重复执行。

**21. MySQL 性能优化你一般怎么做？**

可以这样答：

我会先定位瓶颈，而不是直接加索引。一般看慢 SQL、执行计划、扫描行数、索引命中、锁等待和数据量。

常见优化包括：根据 where、join、order by 建合适的联合索引；避免函数或隐式转换导致索引失效；大分页改成基于游标或延迟关联；热点表按业务维度拆分或归档；批量写入代替单条循环写；必要时做读写分离、缓存或异步化。

但我也会注意索引不是越多越好，写多的表索引过多会影响写入性能。对于任务日志、设备数据这种大表，我会考虑冷热分离、分区、归档或使用更适合的 OLAP / 时序存储。

**22. Redis 你常用在哪些场景？有什么坑？**

可以这样答：

我常用在缓存、分布式锁、热点数据、状态快照、限流和短期幂等记录等场景。

坑主要有几个：缓存穿透、击穿、雪崩；大 key；热 key；锁误释放；缓存和数据库一致性。比如分布式锁要有唯一 value，释放时校验 value，避免释放别人的锁。缓存一致性一般不会追求强一致，而是根据业务选择更新数据库后删除缓存、设置合理过期时间、异步刷新或消息驱动失效。

在机器人状态场景，Redis 可以保存设备最新状态，但历史状态和审计数据不能只放 Redis，最终还是要进入持久化存储。

**23. 你如何保证跨系统数据一致性？**

可以这样答：

我之前在 SCM、EAM、RPA 和 GPT 平台都遇到过跨系统集成，比如 K3、WMS、OMS、PLM、订购系统、模型服务等。我的思路是先区分强一致和最终一致。

如果是本地事务内的数据，尽量用数据库事务保证。跨系统调用时，不轻易做长事务，而是通过状态机、调用记录、重试、补偿和对账保证最终一致。

比如下推 WMS 或 OMS，我会记录第三方调用记录，包括请求参数、响应结果、状态、重试次数和错误信息。业务单据状态不能只依赖接口返回，要有明确的中间状态，比如“下推中、下推成功、下推失败、待重试”。这样失败可恢复，问题可追踪，也方便人工介入。

**25. 如果入职后让你 3 个月内负责一个机器人任务调度模块，你怎么开展？**

可以这样答：

第一个月我会先熟悉业务和现有架构，重点看任务类型、机器人资源模型、状态机、调度策略、失败场景、接口调用链和线上问题。同时梳理现有任务状态和日志是否可追踪。

第三个月开始做抽象和优化，比如统一任务模型、支持优先级、资源匹配、幂等下发、任务取消和恢复机制，同时补齐单元测试、压测和告警指标。我的原则是先理解真实业务，再逐步平台化，不一上来就大重构。

**27. 你在项目中遇到过最大的问题是什么？怎么解决？**

可以这样答：

可以讲一个跨系统集成或状态一致性问题。

比如在 EAM 或 SCM 里，业务流程会涉及多个系统：PLM、OMS、WMS、K3、主数据、审批流等。问题通常不是单个接口开发，而是一个环节失败后，业务单据状态、第三方系统状态和用户看到的状态不一致。

我的处理方式是把外部调用显式建模，增加第三方调用记录和业务状态机。每次调用都记录请求、响应、错误、重试次数和关联单据。业务状态设计成可恢复的中间态，而不是简单成功/失败。对于可重试错误走自动重试，对于业务校验失败给出明确提示，对于长期失败进入人工处理。这样系统从“出错后靠查日志”变成“出错后可追踪、可补偿”。
